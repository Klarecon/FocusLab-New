A1 Accounts & Persistence

Goal: paid plans live in the user's Supabase account, survive a browser clear / device switch, and anonymous users can claim their local plan on signup.

Workstream A1

Accounts & Persistence

A1-T1Stand up Supabase project + schema👤 Mona⏱ ½ day🔴 BLOCKS
Create the Supabase project and the tables that hold users, plans, and entitlements.
  • Create Supabase project; save URL + anon key + service-role key to .env.local and Vercel env (never commit)
  • Add key names to .env.example as empty placeholders (documented, not valued)
  • Create profiles table (id → auth.users, email, created_at)
  • Create plans table (id, user_id FK, payload JSONB = full ParetoResult + Focus Table state, updated_at)
  • Create entitlements table (user_id FK, tier free|monthly|yearly, status, current_period_end)
  • Enable Row Level Security; policies so a user only reads/writes their own rows
  • Commit docs/supabase-schema.sql snapshot (schema only, no secrets)
A1-T2Auth: magic-link (+ optional Google)👤 Mona⏱ 1 day🔴 BLOCKS🔗 A1-T1
Wire Supabase Auth so a user signs in with a magic-link email (Google optional).
  • Install @supabase/supabase-js + @supabase/ssr; create browser + server client helpers
  • Build sign-in screen (email + "send magic link") — pink #c4186a CTA, Fraunces header, Jakarta body
  • Add magic-link callback route that establishes the session cookie
  • Add Google OAuth provider + button (optional; behind a config flag if time-boxed)
  • Add useUser() hook / session context; show signed-in email + sign-out in header
  • Tests: signed-out vs signed-in render states; callback sets session
  • Verify: tsc --noEmit + vitest --run green (no regression on 353)
A1-T3Persist plans to Supabase (replace localStorage as source of truth)👤 Mona⏱ 1 day🔴 BLOCKS🔗 A1-T2
Save/load the Focus Table + Analyzer result to the logged-in user's Supabase row instead of Zustand-persist.
  • Add store actions savePlanToCloud() / loadPlanFromCloud()
  • Debounced autosave on Focus Table edits — Zustand as in-memory cache, Supabase as truth
  • On login, hydrate store from cloud plan if one exists
  • Keep window.__* OUT (hook check #4) — state through the store only
  • Handle offline/error: retry + non-blocking toast, never lose local edits
  • Tests: save writes correct JSONB shape; load rehydrates ParetoResult identically (round-trip)
  • Verify round-trip: edit → clear browser storage → reload → plan restored from cloud
A1-T4"Claim your saved plan" migration for anonymous users👤 Mona⏱ ½ day🔴 BLOCKS🔗 A1-T3
On first signup, migrate the existing local plan into the new account.
  • Detect local plan present + user just authenticated with no cloud plan
  • Show "We found your saved plan — claim it?" prompt (pink CTA)
  • On confirm, upload local payload to plans, mark local as migrated (no re-prompt loop)
  • Conflict rule: if cloud plan exists, keep cloud, offer "replace with local" as explicit choice
  • Tests: migration maps local → cloud identically; no double-migration; conflict path covered
  • Manual QA: anonymous run → build plan → sign up → plan claimed intact
Workstream A2

Billing & Paywall (sandbox first)

A2-T1Lemon Squeezy sandbox setup + products👤 Mona (Oren advises pricing)⏱ ½ day🔴 BLOCKS
Create the sandbox store and three products (Free / $29-mo / $199-yr).
  • Create LS sandbox store; add API key + store ID to .env.local + Vercel env
  • Create variants: Free, Monthly $29, Yearly $199; note variant IDs in config
  • Generate webhook signing secret; store in env
  • Document env var names in .env.example
  • Confirm test-card flow works in sandbox checkout manually (no code yet)
A2-T2Checkout: single-email hosted checkout👤 Mona⏱ ½ day🔴 BLOCKS🔗 A2-T1 · A4-T3 · A1-T2
Launch LS checkout prefilled with the already-captured email — do NOT ask for email again.
  • Read the one captured email from session/store; prefill LS checkout[email]
  • Pass custom data = Supabase user_id so the webhook can map payment → user
  • Build plan selector (Monthly/Yearly toggle) → open hosted checkout — pink CTAs
  • Assert one email source, never re-prompted
  • Tests: checkout URL contains prefilled email + user_id custom field
  • Manual: click buy → LS opens with email prefilled
A2-T3Webhook: payment succeeded → unlock user👤 Mona⏱ 1 day🔴 BLOCKS🔗 A2-T1 · A1-T1
A verified LS webhook flips the user's entitlement to paid.
  • Add /api/webhooks/lemonsqueezy; verify HMAC signature against the webhook secret
  • Handle order/subscription created + payment success → upsert entitlements (tier, active, period_end)
  • Handle cancelled / expired → downgrade to free
  • Idempotency: dedupe by LS event/order id (no double-unlock)
  • Map payment → user via custom.user_id; fallback match by email
  • Tests: signed payload unlocks; bad signature 401; duplicate no-ops; cancel downgrades
  • Verify: sandbox purchase → entitlement goes active within seconds
A2-T4Paywall gate on the Focus Table👤 Mona⏱ ½ day🔴 BLOCKS🔗 A2-T3 · A1-T2
Gate /focus behind entitlement; free users see the paywall, paid users pass. Paywall ONLY here — never on /analyzer.
  • Server-check entitlement before rendering /focus; unpaid → paywall screen
  • Paywall screen: value recap + Monthly/Yearly buttons (→ A2-T2), pink CTA, human emoji
  • Paid user reaching /focus renders the table immediately
  • Real-time/poll unlock so the just-paid user advances without manual refresh
  • Tests: unpaid blocked, paid allowed, entitlement flip re-renders access
  • Verify: pre-commit hook passes all 11 checks
A2-T5End-to-end sandbox proof + live-swap runbook👤 Mona (exec) · Oren (live acct)⏱ ½ day🔴 BLOCKS🔗 A2-T2/T3/T4 · A1-T3
Prove stranger → pay → unlock end-to-end, and write the 30-min live-account swap steps. This is KR 1.1.
  • Full run on deployed preview: fresh browser → analyzer → email gate → checkout (test card) → webhook → /focus unlocked
  • Confirm plan persists to that account (A1) after payment
  • Screen-record the passing run (evidence for KR 1.1)
  • Write docs/go-live-lemonsqueezy.md: swap sandbox→live keys, live webhook URL, live variant IDs, verify one real transaction
  • Add Playwright coverage of the gate transition where feasible (mock webhook unlock)
  • Oren: provide live LS store + API key/webhook secret at launch (30-min plug-in, final step)
